Architecture — revised

LLM summaries capture reasoning only; facts are gathered by the orchestrator

Stage output: two channels
LLM produces:
✦ reasoning & decisions
✦ ambiguities encountered
✦ scope changes identified
✦ anything that surprised the LLM

Orchestrator gathers:
● git diff / file changes
● test results (pass/fail)
● comments posted (via GitHub API)
● PR URL, branch name, labels
● any factual, observable state

Combined for next stage:
{summaries + gathered_facts}
Example: implementation stage
LLM summary:
"Needed to refactor UserService
to support async — issue didn't mention
this. Chose callback pattern over
promises for compat with existing code."

Orchestrator gathers:
"3 files changed, 47 insertions, 12 deletions.
Tests: 8 passed, 0 failed.
Branch: fix/issue-42-async-user."

Next stage gets both — reasoning + facts.

Principle: LLM tells you why, orchestrator tells you what

The LLM's structured summary is strictly for things that require judgment. Anything that can be observed programmatically — file diffs, test results, GitHub state — is the orchestrator's job to collect and pass forward. This keeps LLM output small and prevents the LLM from inaccurately summarizing facts the orchestrator can verify directly.
