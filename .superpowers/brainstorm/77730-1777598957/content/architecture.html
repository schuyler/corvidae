Proposed architecture

How the pieces fit together

Deterministic Orchestrator (Python)
Workflow definition:
[comment] → [label] → [branch] →
[gate: ready?] → [red] → [green] → [refactor] →
[gate: docs?] → [docs] → [PR] →
[feedback?] → [respond] → [close]

Each stage:
1. Build prompt (system + stage instruction + prior summaries)
2. Invoke LLM via Corvidae's run_agent_turn
3. Execute tools (shell, GitHub API, file I/O)
4. Verify exit criteria
5. Optionally gate for human approval
6. Produce structured summary for next stages
7. Determine next stage (forward or escape hatch)
LLM Invocation per Stage
Input to LLM:
system: "You are implementing stage X..."
system: "Prior context: {summaries}"
system: "Available tools: {stage tools}"
user: "{stage-specific instruction}"

LLM does its work:
→ Calls tools (write_file, shell, github API...)
→ Calls stage_complete(outcome, summary)

Output from LLM:
{outcome: "proceed", summary: "..."}
{outcome: "need_clarification", summary: "..."}
{outcome: "scope_changed", summary: "..."}

Key design properties

Stage isolation

Each LLM call starts fresh — no accumulated context drift. Only prior summaries flow forward.

Tool scoping

Each stage only gets the tools it needs. The implementation stage gets shell/file tools; the PR stage gets GitHub API tools.

Deterministic flow

The orchestrator controls transitions. The LLM can't skip stages — it can only signal outcomes within the current stage.

Human checkpoints

Configurable per stage. The orchestrator pauses and waits for human approval before proceeding.
