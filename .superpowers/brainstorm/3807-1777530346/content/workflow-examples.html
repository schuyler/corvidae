Three concrete scenarios

Let's trace the workflow through these to see what shape emerges

Scenario A: Clean implementation

A well-written issue for a small feature. No ambiguity, no PR feedback.

Happy path — linear
● comment on issue → "Planning to implement X"
● add "in progress" label
● create branch & worktree
● gate: ready to implement? → yes
● write tests (red)
● implement (green)
● refactor
● gate: docs needed? → no
● submit PR
○ respond to feedback — skipped, PR approved
● close issue

Scenario B: Ambiguous issue, PR feedback

An issue with unclear requirements. PR gets review feedback requiring changes.

Messy path — loops back
● comment on issue → "What does X mean in context Y?"
● add "in progress" label
● create branch & worktree
gate: ready to implement? → no, need clarification
● comment on issue → more questions
gate: ready to implement? → no
● comment on issue → "Also, should Z be affected?"
gate: ready to implement? → yes
● red / green / refactor
● submit PR
● feedback received → "Needs error handling"
● respond to feedback → add error handling, push
● close issue

Scenario C: Implementation reveals scope change

While coding, the agent discovers the issue affects more than expected.

Discovery path — back to planning
● comment / label / branch
gate: ready to implement? → yes
● write tests (red)
● implement (green) → discovers dependency on module Q
gate: still on track? → no, scope expanded
● comment on issue → "This also affects Q. OK to expand scope?"
gate: ready to implement? → yes (updated scope)
● red / green / refactor (expanded)
● gate: docs needed? → yes
● update docs
● submit PR → close issue

What shape do these suggest?

A

Linear with gates and explicit retry targets

Each gate can loop back to a specific earlier stage. The orchestrator defines which loops are allowed.

B

General state machine

Each stage is a state. Transitions are determined by LLM gate decisions. Any transition is possible.

C

Hybrid: linear backbone with escape hatches

A default forward path, but specific stages can "escape" to earlier points when conditions warrant.
